# identify_excludable_code_candidates Tool

## Overview

The `identify_excludable_code_candidates` tool identifies code that should be excluded from coverage metrics. It uses conservative detection criteria to minimize false positives and ensure only truly excludable code is flagged.

This tool leverages the `ExcludableCodeCandidateService` to provide intelligent exclusion candidates with confidence scores and detailed analysis.

## Function Signature

```python
def identify_excludable_code_candidates(
    repo_slug: str,
    provider: str,
    ref: str,
) -> Dict[str, Any]
```

## Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `repo_slug` | `str` | Yes | Repository slug in format `owner/repo` |
| `provider` | `str` | Yes | Coverage provider name (e.g., `codecov`, `coveralls`) |
| `ref` | `str` | Yes | Git reference (branch, tag, or commit SHA) |

## Return Value

Returns a dictionary with the following structure:

```python
{
    "ref": str,                           # Git reference analyzed
    "total_files": int,                   # Total files analyzed
    "candidate_count": int,               # Number of excludable candidates
    "candidates": [                       # List of excludable code candidates
        {
            "file_path": str,             # Path to the file
            "excludability_score": float, # Excludability score (0-1)
            "should_exclude": bool,       # Whether to exclude from coverage
            "breakdown": {                # Breakdown of detection signals
                "generated": {...},       # Generated code analysis
                "vendor": {...},          # Vendor code analysis
                "migration": {...},       # Migration/deprecated analysis
                "zero_hit": {...},        # Zero-hit signal analysis
            },
        },
        ...
    ],
    "summary": str,                       # Summary of findings
}
```

## Error Responses

The tool returns error responses for invalid inputs:

```python
{
    "error": str,           # Error message
    "error_code": str,      # Error code (INVALID_REPO_SLUG)
}
```

## Usage Examples

### Identify excludable code candidates for main branch

```python
result = identify_excludable_code_candidates(
    repo_slug="myorg/myapp",
    provider="codecov",
    ref="main",
)
```

### Identify excludable code candidates for a specific commit

## Output Interpretation

### Excludability Score

The `excludability_score` (0-1) indicates how likely the code is to be excludable:

- **0.0-0.3**: Insufficient evidence, likely not excludable
- **0.3-0.5**: Weak signal, needs manual review
- **0.5-0.7**: Moderate confidence, likely excludable
- **0.7-0.9**: High confidence, probably excludable
- **0.9-1.0**: Very high confidence, definitely excludable

### Detection Signals

The `breakdown` field contains analysis for each detection type:

#### Generated Code

Indicators:
- Explicit markers: `AUTO-GENERATED`, `DO NOT EDIT`, `GENERATED CODE`
- File extensions: `.pb2.py`, `.pb2_grpc.py`, `.g.dart`
- Generation patterns: "generated by", "generated from"

#### Vendor Code

Indicators:
- Vendor directories: `vendor/`, `node_modules/`, `.venv/`, `site-packages/`
- Third-party paths: `third_party/`, `external/`
- Common vendor files: `jquery.js`, `bootstrap.js`, `react.js`

#### Migration/Deprecated Code

Indicators:
- Deprecation markers: `DEPRECATED`, `LEGACY`
- Migration comments: "will be removed", "to be removed"
- Version-specific paths: `v1/`, `v2/`

#### Zero-Hit Signals

Indicators:
- File has zero hits
- All lines are uncovered
- Unreachable code patterns: `if False:`, `raise NotImplementedError`

## Best Practices

1. **Review high-confidence candidates**: Start with candidates with score > 0.7
2. **Verify before excluding**: Always verify candidates before adding to exclusions
3. **Document decisions**: Document why code is excluded
4. **Use conservative approach**: Only exclude when confident
5. **Regular audits**: Run this tool regularly to find new candidates

## Conservative Detection Approach

The tool uses strict criteria to avoid false positives:

### Disqualifying Factors

Code is NOT excluded if it:

1. **Defines Public API**
   - Has `__all__` definition
   - Contains public classes/functions

2. **Has Side Effects**
   - Modifies `sys.path`
   - Accesses `os.environ`
   - Performs monkey patching
   - Modifies global state

3. **Is Test Code**
   - Located in test directories
   - Named with test patterns

## Common Excludable Code Types

### Generated Code

```python
# AUTO-GENERATED
# DO NOT EDIT
# Generated by protoc

class GeneratedMessage:
    pass
```

### Vendor Code

```
node_modules/package/index.js
.venv/lib/python/site-packages/module.py
vendor/lib/module.py
```

### Deprecated Code

```python
# DEPRECATED - Use new_function instead
# TODO: remove in v2.0
def old_function():
    pass
```

### Unreachable Code

```python
if False:
    def unreachable():
        pass

raise NotImplementedError("Not yet implemented")
```
